# Mini-12306 项目计划书

## 1. 项目背景

Mini-12306 是一个简化版中国铁路 12306 在线车票服务系统。项目目标是用互联网软件替代传统售票厅排队模式，为旅客提供车次查询、购票、退票、改签等核心能力，并为管理员提供系统参数配置能力。

本项目为课程作业，不接入真实铁路、支付、身份证认证或短信服务。相关第三方能力通过模拟服务完成，重点体现需求分析、系统设计、数据库设计、前后端工程实现、测试和文档规范。

## 2. 项目目标

- 旅客可以实名注册、登录并维护基本身份信息。
- 旅客可以按日期、出发站、到达站查询可售车次和余票。
- 旅客可以在线下单、模拟支付并生成车票。
- 旅客可以对符合条件的车票发起退票。
- 旅客可以对符合条件的车票发起改签，并处理差价支付或退款记录。
- 管理员可以配置系统参数，并维护基础车次、站点、经停和票额数据。
- 系统具有清晰的前后端分层、API 契约、数据库约束、测试用例和可演示数据。

## 3. 范围边界

### 必做范围

- 用户注册、登录、退出、当前用户信息查询。
- 站点列表、车次查询、车次详情、余票展示。
- 订单创建、订单支付、订单取消、订单详情。
- 车票列表、车票详情、退票、改签。
- 管理员系统设置、站点管理、车次管理、经停管理、票额管理。
- 身份证认证、支付、短信通知的模拟实现。

### 不做范围

- 不实现真实 12306 复杂候补、抢票、积分、会员、保险、发票、选座图、跨境车票等能力。
- 不接入真实银行、支付平台、公安身份认证或短信网关。
- 不追求高并发抢票级别架构，但核心库存扣减必须正确。
- 不开发原生 App，前端以 Web 页面为主。

## 4. 用户角色

- 游客：浏览登录页、注册页、公开车次查询入口。
- 旅客：注册登录后查询车次、购票、支付、退票、改签、查看订单和车票。
- 售票员：课程需求中提及用户端可面向售票员；本项目可作为增强角色，默认不进入 MVP。
- 管理员：维护站点、车次、经停、票额、系统参数，查看基础运营数据。

## 6. 推荐里程碑

### M1：项目骨架与规范

- 建立前后端目录、环境配置、代码格式化、基础 README。
- 建立数据库连接、迁移机制、健康检查接口。
- 建立 Vue 路由、布局、登录态管理、API 客户端。
- 输出需求、架构、API、数据库、测试规范。

### M2：账号与基础数据

- 完成用户注册、登录、角色鉴权。
- 完成站点、车次、经停、票额模型和管理接口。
- 准备演示数据：站点、车次、经停、不同席别余票。

### M3：查询与购票闭环

- 完成车次查询接口和前端查询页。
- 完成下单、库存锁定、支付模拟、出票。
- 完成订单列表、订单详情、车票列表。

### M4：退票与改签

- 完成退票状态流转、库存释放、退款记录。
- 完成改签目标车次查询、差价计算、事务处理。
- 完成前端退票和改签操作流程。

### M5：质量收口与答辩材料

- 补齐核心单元测试、接口测试、前端关键流程测试。
- 完成部署说明、测试报告、数据库脚本、演示账号。
- 准备项目演示脚本和答辩说明。

## 7. 交付物

- 前端源码：Vue 3 + TailwindCSS Web 应用。
- 后端源码：Go + Gin + GORM 服务。
- 数据库脚本：建表、索引、初始化数据、迁移记录。
- 接口文档：统一响应、错误码、核心接口请求响应示例。
- 测试文档：测试范围、测试用例、测试结果、已知问题。
- 项目说明：运行方式、环境变量、演示账号、功能截图。

## 8. 验收标准

- 可以从零启动 MySQL、后端和前端，并完成完整购票流程。
- 查询车次时余票、票价、出发到达时间展示正确。
- 支付成功后生成车票，库存减少；取消或超时订单释放库存。
- 退票成功后车票状态变更，库存恢复，生成退款记录。
- 改签成功后原票关闭，新票生成，库存和差价记录正确。
- 管理员接口受权限控制，普通旅客不能访问。
- 核心写操作具备事务保护，重复提交不会造成重复扣库存或重复出票。

## 9. 主要风险与应对

- 时间风险：优先做 P0 垂直闭环，再做管理后台增强。
- 质量风险：先定状态机、接口契约和数据库约束，再写页面。
- 技术风险：后端优先保证事务和库存正确；前端优先保证表单、状态和错误提示清晰。
- 演示风险：准备固定演示数据和演示账号，避免现场临时造数据。
